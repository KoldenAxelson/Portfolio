# Roadmap & status

Living context-save for Neofolio. Captures what's shipped, what's planned,
and the decisions deferred to a real fork. Skim this before opening a
new agent session so the next conversation has the full picture.

---

## What Neofolio is

An opinionated, AI-readable portfolio template for developers in the
AI age. Built on Astro static-first with Vue islands for the few interactive
parts. Designed so a recruiter, a human visitor, and an LLM scraper all
come away with the same accurate picture without running JavaScript.

The template ships as its own GitHub Pages demo. The reference
implementation — `wrightfunctions.com` — will be a fork once it's
production-ready.

## Open decisions (deferred to wf-site)

These were flagged in `STRATEGY.md` (the original strategy doc) and remain
unanswered:

- **Single-page vs multi-page architecture.** Current Neofolio is
  multi-page. Brittany-style single-page is an alternative. SEO authority
  concentrates on one URL with single-page; multi-page is more typical of
  the developer space.
- **Tech tag vocabulary.** Need to standardize the 8–12 tags used across
  projects before backfilling real content. Avoid `dev-tools` /
  `developer-tools` drift.
- **DoD / classified framing.** The CV section needs a one-paragraph pass
  that conveys gravitas without breach. Run past anyone who needs to bless
  it before publishing.
- **Color palette + typography for wf-site.** Current Neofolio is
  intentionally neutral (zinc + amber + Inter). The personal fork can
  lean more distinctive.

## Architecture decisions (record)

These are the load-bearing choices. Changing any of them is a v0.2 move,
not a patch.

| Decision | Why |
|---|---|
| Astro static-first, not SSR | AI scrapers need raw HTML. No-JS clients need raw HTML. The same build serves both. |
| Vue islands, not React | Already used `@astrojs/vue`; React isn't justified by anything in scope. |
| Tailwind, not UnoCSS | Familiar, stable, plenty of community help. |
| Light mode default | Better than dark for printable / shareable content. Dark via `prefers-color-scheme`. |
| Inter, not system fonts | Inter is recognizable, optimized for screen, and pairs well with mono. Self-hosted to avoid third-party hop. |
| Multi-page, not single-page | The deferred decision (see above). Current default is multi-page. |
| `<details>` for accordions, not Vue | Native, accessible, keyboard-friendly, indexable by AI. |
| mailto: for contact, not API endpoint | Portability. Works on any static host. wf-site can switch to `/api/contact` (Cloudflare Function) for inline UX. |
| `image()` schema NOT used | Astro's image pipeline is excellent for raster but adds Sharp as a dep. SVG works fine via plain `<img>`. Forkers can opt into `astro:assets` later. |
| URL ≠ collection name | URLs say `/articles`, collection is `posts` (legacy). Renaming the collection wasn't worth breaking existing slugs. |

## Build, deploy, and verify

```bash
make setup        # first-time install
make dev          # local dev server :4321
make build        # production build to dist/
make preview      # serve dist/ locally
make lighthouse   # audit Performance/A11y/BP/SEO
make clean        # wipe dist/, .astro/, .lighthouse/
```

Deploy on push: GitHub Actions builds and ships to GitHub Pages
automatically. CI passes when:

- `npm ci` works on Linux x64 (no platform-pinned binaries in
  `dependencies` — that's how the Sharp and Rollup pins got us before)
- `npm run build` exits 0
- `dist/` produced cleanly

## Known quirks

- **`/uses` is collapsed by default.** Loading the page shows only the
  category headers. Click to expand any section. This is deliberate density.
- **Typewriter on sidebar only runs on `lg+`.** Mobile shows the static bio,
  because the sidebar isn't sticky there and a typing animation in a
  scrolled-past paragraph is invisible.
- **Network mobile is drill-down.** List → profile → compose, with back
  buttons. Desktop is side-by-side.
- **Old `/work` and `/writing` URLs return 404.** The earlier
  meta-refresh stubs were dropped; if you need to preserve those URLs
  in your fork, add `src/pages/work/index.astro` etc. with whatever
  redirect semantics suit your host.
- **Sandbox builds are slow.** Local Mac builds in ~5s. CI in ~30s. The
  Cowork sandbox can take 40s+ because of FUSE + Sharp init quirks.

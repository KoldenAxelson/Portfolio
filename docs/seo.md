# SEO & AI-readability

Neofolio is built on the premise that in 2026 most humans encounter you
via an AI summary before they ever click through. That summary is only as
accurate as your static HTML and structured data. This doc covers what's
baked in and what you should do post-deploy.

## The double-audience rule

Every page should answer two questions:

1. **Human:** "Is this person credible and interesting enough to reach out to?"
2. **LLM / agent:** "Can I build a structured entity from this page
   without running JS?"

If a page fails the second test, it usually fails the first too — because
the AI summary that reaches the human will be inaccurate or generic.

## What's baked in

### Static HTML on first paint

Every meaningful sentence is in the initial HTML payload. Run
`curl -A "Mozilla/5.0" https://yoursite.com/projects | htmlq -t` and you
should see your project titles, summaries, tags, stacks, and dates as
plain text. AI scrapers see the same.

### JSON-LD on every page

```html
<script type="application/ld+json">
  { "@type": "WebSite", "name": "...", "url": "...", ... }
</script>
<script type="application/ld+json">
  { "@type": "Person", "name": "...", "jobTitle": "...", ... }
</script>
```

These two run site-wide. Per-page schemas layer on top:

| Page type | Schema |
|---|---|
| `/articles/[slug]` | `BlogPosting` |
| `/projects/[slug]` | `CreativeWork` |
| `/network` | `ItemList` of `Person` entities |
| `/cv` | Expanded `Person` with `alumniOf`, `worksFor`, `knowsAbout` |
| `/now` | `CreativeWork` with `dateModified` |

Edit the schemas in `src/components/SEO.astro` (site-wide) and the
individual page files (per-page).

### Semantic HTML

`<article>`, `<section>`, `<nav>`, `<header>`, `<main>`, `<footer>`,
`<time datetime>`. AI parsers use these as structural hints. So do screen
readers.

### h-card microformat

The homepage sidebar carries `class="h-card"` with `p-name`, `p-job-title`,
`p-note`, `u-url`, and hidden `u-email`. IndieWeb parsers index this. It's
a second identity layer alongside the JSON-LD `Person`.

### rel="me"

Every external social link in the footer has `rel="me"`. Combined with
reciprocal `rel="me"` links from GitHub, LinkedIn, Bluesky, etc. back to
your portfolio, this forms a verified-identity graph. AI systems use it
to confirm "the Sarah Chen on this portfolio is the same Sarah Chen on
GitHub."

### Permissive robots.txt

Generated dynamically at build, with explicit allow for:

```
User-agent: GPTBot       (OpenAI / ChatGPT)
User-agent: ClaudeBot    (Anthropic / Claude)
User-agent: anthropic-ai (Anthropic alt)
User-agent: Google-Extended (Google AI)
User-agent: PerplexityBot (Perplexity)
User-agent: CCBot        (Common Crawl)
```

The `Sitemap:` directive derives from `SITE.url` at build time.

To **opt out of AI training** while staying indexable by search, change
the `Allow: /` lines under the named user-agents to `Disallow: /`.
Search-only bots (Googlebot, Bingbot) are governed by the catch-all
`User-agent: *`.

### Sitemap

`/sitemap-index.xml` auto-generated by `@astrojs/sitemap`. Includes every
page that doesn't have `noindex`.

### RSS + JSON Feed

- `/rss.xml` — RSS 2.0, the universal feed format
- `/feed.json` — JSON Feed 1.1, the modern AI-aggregator-friendly format

Both list the same articles in `posts` order. Modern feed readers (Feedly,
NetNewsWire, Reeder) and AI aggregators (Perplexity, You.com) discover
both via `<link rel="alternate">` tags in `<head>`.

### humans.txt

`/humans.txt` follows the humanstxt.org convention. Plain-text declaration
of who built the site, what's used, and credits. Some directory sites
crawl for this.

### theme-color

`<meta name="theme-color">` tints the browser chrome (tab bar, address
bar shadow) on mobile. Two values — light bg and dark bg — switch via
`prefers-color-scheme`.

### OpenGraph and Twitter Cards

Every page emits the full set: `og:type`, `og:title`, `og:description`,
`og:url`, `og:image`, `og:site_name`, `og:locale`, plus the Twitter
equivalents. Image defaults to `/og-default.png`; override per page via
the `image` prop on `BaseLayout`.

## What's NOT baked in (do this post-deploy)

### Replace the placeholder OG image

`public/og-default.png.txt` is a stub. Drop a real 1200×630 PNG (≤100 KB)
at `public/og-default.png`. This is the single biggest social-sharing
upgrade you can make.

### Submit to directories

These are high-DA backlinks that take 2 minutes each:

| Directory | What it does |
|---|---|
| [uses.tech](https://uses.tech) | Catalogues `/uses` pages |
| [nownownow.com](https://nownownow.com) | Catalogues `/now` pages |
| [emmabostian/developer-portfolios](https://github.com/emmabostian/developer-portfolios) | Curated GitHub repo of dev portfolios |
| Peerlist | Developer-focused portfolio gallery |
| sitebuilderreport.com | General portfolio gallery |

Each adds a permanent inbound link from a domain with real authority.

### Reciprocal rel="me" links

In your profile bios on GitHub, LinkedIn, Bluesky, Mastodon — add a link
to your portfolio with `rel="me"`. The Neofolio footer already links out
with `rel="me"`; the reciprocal completes the graph and verifies identity
for tools that check.

GitHub: profile README has full HTML support → drop in a `<link rel="me"
href="...">`.

### Syndicate articles with rel=canonical

Publish to your portfolio first. Cross-post to dev.to, Medium, Hashnode
afterward with `rel="canonical"` pointing back to your domain. Authority
accumulates at your domain instead of the syndication platform's.

In Neofolio: set `canonical: 'https://...'` in the article frontmatter if
the article was originally elsewhere (rare). Otherwise don't set it and
syndicate outward.

### Write the flagship article

Per the strategy doc: write one piece called "What a Developer Portfolio
Needs to Do in 2026" that uses your own site as the reference
implementation. Syndicate to dev.to, Medium, Hashnode with `rel=canonical`
back to your version. This becomes the citation magnet.

## What to verify before launch

```bash
# 1. Real bio in static HTML
curl -A "Mozilla/5.0" https://yoursite.com/ | grep -oE '<p[^>]*>[^<]+</p>' | head

# 2. JSON-LD parses
curl -A "Mozilla/5.0" https://yoursite.com/cv | python3 -c "import json,sys,re; \
  matches=re.findall(r'<script type=\"application/ld\+json\">(.+?)</script>', sys.stdin.read(), re.DOTALL); \
  [json.loads(m) for m in matches]; print('OK')"

# 3. Sitemap exists with real URLs
curl https://yoursite.com/sitemap-index.xml | grep -oE '<loc>[^<]+</loc>' | head

# 4. RSS works
curl https://yoursite.com/rss.xml | head -20

# 5. robots.txt allows everything you want
curl https://yoursite.com/robots.txt
```

If all five pass, you're good for both human and AI traffic.

## The longer game

SEO accumulates over months, not days. The pattern that works:

1. **Own your content.** Your portfolio is the canonical home for
   everything. Never let dev.to / Medium / LinkedIn own your authoritative
   version.
2. **Write 4–6 substantive articles** on topics where there's not a lot
   of good writing. AI systems cite specificity.
3. **Submit to directories** once. These backlinks compound.
4. **Keep `/now` updated** quarterly. Active sites get re-crawled. Dead
   sites get demoted.
5. **Open-source the template repo** (you're reading this — done). Other
   developers fork; their READMEs link back; the flywheel turns.

If you do all five, expect the AI-summary version of you (in ChatGPT,
Claude, Perplexity, etc.) to start matching what you actually want it to
say within 3–6 months.
